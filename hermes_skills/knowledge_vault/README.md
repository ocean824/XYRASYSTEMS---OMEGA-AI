# Skill: `knowledge_vault`

> **Primary callers:** Α — LOOM (Memory & Continuity Engine), Υ — ARCHIVE (Data Storage & Structuring).
> **Engine reference:** Open-source `hermes-agent` from Nous Research — the self-improving CLI agent that writes reusable skill artifacts from successful tasks.
> **Purpose:** A standardized self-learning ingestion pipeline. The operator drops domain knowledge (PDFs, transcripts, screenshots, Markdown notes) into a watched directory; LOOM parses and routes it to the right downstream agents; ARCHIVE persists the structured index; `hermes-agent` writes the reusable skills that emerge.

## Why It Exists

The canonical 25 agents already cover memory (LOOM) and storage (ARCHIVE). What was missing was a *standard contract* for how new domain knowledge enters the system without requiring code changes. This skill formalizes that contract so any operator can extend any vertical's competence by file-drop.

## Operator Workflow

```
1. Operator drops a file into <deployment>/knowledge_vault/inbox/
   Supported: .md, .pdf, .txt, .png/.jpg (OCR), .mp3/.mp4 (transcribed via Sonus or
   manus-speech-to-text), .json, .csv
2. LOOM file-watcher fires within seconds.
3. LOOM hands off to a parser pipeline:
     - text artifacts → chunk by semantic boundary
     - images → OCR + caption
     - audio/video → transcribe + chunk
4. Each chunk is tagged with:
     - vertical (trading | marketing | engineering | outreach | other)
     - source_creator (e.g., "MoonDev", "Wyckoff classical", "internal SOP")
     - methodology (e.g., "order_flow", "wyckoff", "auction_theory", "n/a")
     - confidence (operator-provided or default 0.5)
5. ARCHIVE persists the chunks with embedding vectors.
6. `hermes-agent` (running as a subordinate skill, NOT as a 26th agent) reviews
   the new corpus and proposes new reusable skill artifacts (Python helpers,
   prompt templates, n8n workflows, indicator code).
7. Proposed skills go through the validation_gauntlet before any agent uses them.
```

## Folder Layout (per deployment)

```
knowledge_vault/
├── inbox/                  ← operator drops files here
├── parsed/                 ← chunked & tagged artifacts (immutable after write)
├── proposed_skills/        ← hermes-agent's drafts pending validation_gauntlet
├── promoted_skills/        ← gauntlet-passed skills now callable by the 25 agents
└── quarantine/             ← gauntlet-failed or operator-rejected artifacts
```

## Contract

### Input — `IngestRequest` (auto-generated by file-watcher)
```yaml
artifact_path: knowledge_vault/inbox/<filename>
operator_metadata:        # optional — operator can pre-tag at drop time via sidecar .yaml
  vertical: <enum>
  source_creator: "<string>"
  methodology: "<string>"
  confidence: 0.0–1.0
```

### Output — `IngestResult`
```yaml
ingest_id: <uuid>
artifact_path: <path>
status: ingested | failed | quarantined
chunks_emitted: <int>
proposed_skills: <list of paths under knowledge_vault/proposed_skills/>
errors: <list of named reasons if any>
```

## Connection to `validation_gauntlet`

Every artifact that emerges as a *proposed skill* must pass `validation_gauntlet` before it is moved from `proposed_skills/` to `promoted_skills/`. This honors ØMEGA's "no speculative writes" anti-pattern: a skill is never marked usable based on ingestion alone.

## Hard Rules

- Files in `parsed/` and `promoted_skills/` are **read-only** once written (`chmod 444` on Linux/macOS; `attrib +R` on Windows). Re-ingestion overwrites with a new versioned file rather than mutating in place.
- `hermes-agent` runs **without** authority to call `MEGALODON`, `TITHE`, `OMNIPOST`, `VANGUARD`, or any other agent that performs irreversible actions. Its job is purely to draft proposed skills for the validation gauntlet.
- Every ingest event is logged to ARCHIVE with the artifact's SHA-256 so provenance can be audited.
